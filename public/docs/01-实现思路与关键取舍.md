# 实现思路与关键取舍

## 整体架构设计

Atoms Demo 采用纯前端架构，基于 Next.js 14 + React 18 + TypeScript + Tailwind CSS 构建。

### 技术选型依据

1. **Next.js (Static Export)**
   - 选择原因：支持静态导出，可部署到任何静态托管服务（Vercel、Netlify、GitHub Pages）
   - 取舍：放弃了 SSR 能力，换取更简单的部署流程

2. **纯前端实现**
   - 选择原因：降低部署复杂度，用户可直接在浏览器中运行
   - 取舍：使用 localStorage 替代后端数据库，数据仅保存在本地

3. **模拟 Agent 协作**
   - 选择原因：真实的多 Agent 系统需要 LLM API 和复杂的编排逻辑
   - 取舍：使用预设模板 + 模拟消息流来展示 Agent 协作过程

## 关键技术难点

### 1. 代码实时预览
- 方案：使用 iframe + document.write 动态注入 HTML/CSS/JS
- 安全：使用 sandbox="allow-scripts" 限制 iframe 权限

### 2. 数据持久化
- 方案：localStorage 存储用户会话和生成历史
- 限制：数据仅在当前浏览器可用，清除缓存会丢失

### 3. Agent 消息流模拟
- 方案：基于 Promise 的异步队列，模拟真实 Agent 的思考和执行过程
- 每个 Agent 有预设的延迟和消息序列

## 功能取舍

| 功能 | 实现方式 | 取舍说明 |
|------|---------|---------|
| 真实 LLM 调用 | 未实现 | 需要 API Key 和费用，使用模板替代 |
| 后端服务 | 未实现 | 纯前端部署，使用 localStorage |
| 多模型 Race 模式 | 未实现 | 复杂度较高，仅展示 UI |
| 真实部署 | 模拟 | 生成可下载的 HTML 文件 |

## 系统架构图

```
用户输入 → AgentEngine → 模板匹配 → 代码生成 → 预览/下载
              ↓
         模拟消息流（多 Agent 协作展示）
              ↓
         localStorage 持久化
```
